## AI 辅助学习规划与复盘系统 - 产品需求文档 (PRD)

### 一、文档定位

本文档是“知音”AI 辅助学习规划与复盘系统的完整愿景版 PRD，用于统一产品方向、客户端页面职责、前后端边界与阶段规划。后续实际开发时，每个阶段应单独拆分 PRD Plan 或迭代任务，不应将本文档中的全部愿景理解为一次性交付范围。

当前仓库是 HarmonyOS NEXT 客户端工程，仅承载 ArkTS/ArkUI 客户端代码。服务端将由独立 Java Spring Boot 工程实现，本文档只描述客户端依赖的后端能力边界，不定义具体 REST URL。

---

### 二、产品概述

#### 2.1 产品定位

本产品是一款 AI 辅助的学习计划、执行跟踪与知识复盘系统。系统结合大语言模型（LLM）、用户学习目标、每日任务状态以及 Git 仓库中的学习笔记或代码提交记录，帮助用户形成“目标设定 -> AI 计划 -> 当日执行 -> Git 复盘 -> 次日调整”的持续学习闭环。

产品强调“辅助决策与复盘”，而不是完全替代用户管理学习过程。AI 可以生成计划、分析偏差、提出建议，但涉及用户知识库写入、仓库提交、长期目标调整等高风险操作时，必须保留用户确认机制。

#### 2.2 目标用户

* 有明确阶段性学习目标的学习者，例如准备一个月内掌握 HarmonyOS 原生应用开发。
* 使用 GitHub、Gitee 或其他 Git 平台管理学习笔记、代码练习或技术总结的用户。
* 希望通过 AI 获得计划拆解、每日提醒、执行复盘和知识盲区提示的自驱型学习者。

#### 2.3 核心价值

* 将长期学习目标拆解成每日可执行任务，降低规划成本。
* 结合 Git 提交记录分析真实学习行为，避免只记录计划、不验证执行。
* 通过 AI 复盘识别遗漏知识点、执行偏差和笔记质量问题。
* 根据复盘结果动态调整后续计划，形成持续优化的学习节奏。

### 三、系统边界与技术架构

#### 3.1 客户端工程边界

HarmonyOS NEXT 客户端采用 ArkTS 与 ArkUI 声明式框架开发，负责：

* 页面展示与用户交互。
* 手机号验证码登录流程的客户端输入与状态展示。
* 初始化向导、今日计划、AI 导师、知识复盘、系统设置等页面。
* 本地轻量缓存，例如登录态、最近一次页面数据、草稿输入内容。
* 调用后端接口并展示结果。

客户端不得直接持有或调用以下敏感能力：

* LLM API Key。
* 短信服务 Access Key。
* Git 平台服务端密钥。
* 可直接写入用户仓库的长期敏感凭证明文。

#### 3.2 服务端工程边界

Java Spring Boot 后端在独立工程中实现，负责：

* 用户认证、登录态签发与权限校验。
* LLM 调用、Prompt 管理、结构化结果校验与失败重试。
* Git 平台授权、仓库同步、Commit/Diff 拉取与分析。
* 每日计划生成、任务状态持久化与复盘报告持久化。
* 短信验证码、学习提醒、定时任务调度。
* 敏感凭证的加密存储、轮换与失效处理。

#### 3.3 第三方服务依赖

* 大语言模型 API：用于计划生成、复盘分析、对话式答疑和计划调整建议。
* Git 平台 API：用于读取仓库信息、提交记录、文件 Diff，并在远期能力中生成修改建议或 PR。
* 短信服务 API：用于手机号验证码与学习提醒触达。

### 四、客户端功能与页面结构

客户端以底部导航栏 Tabs 为主结构。新用户首次登录后先进入初始化向导，完成基础配置后进入主页面。

#### 4.1 启动与引导模块 (Onboarding)

**用户目标**

完成身份确认、学习目标录入和 Git 仓库配置，使系统具备生成计划与复盘分析的基础上下文。

**功能需求**

* 手机号验证码登录：用户输入手机号，获取验证码，完成登录。
* 初始化向导：首次登录强制进入，已完成初始化的用户不再强制展示。
* 学习目标录入：用户填写宏观目标、期望周期、每日可投入时间。
* Git 仓库配置：用户填写仓库地址，选择平台类型，提交 Access Token 或授权结果。
* 授权状态展示：展示仓库是否可访问、默认分支、最近一次校验时间。

**依赖的后端能力**

* 用户与认证能力。
* 学习目标与用户配置能力。
* Git 仓库授权与可访问性校验能力。

#### 4.2 今日计划 (Today Tab)

**用户目标**

快速查看今日学习任务，标记执行状态，并在需要时提前结束学习进入复盘。

**功能需求**

* 顶部状态栏：展示连续打卡天数、今日计划状态、AI 生成的每日鼓励语。
* 今日任务流：展示 3-5 个 AI 拆解出的任务，包括标题、预计时长、状态。
* 任务状态交互：支持将任务标记为待开始、进行中、已完成或跳过。
* 提前复盘入口：用户可手动触发“提前结束今日学习并开始复盘”。
* 计划更新时间：展示今日计划生成时间和最后一次调整时间。

**依赖的后端能力**

* 今日计划生成与查询能力。
* 每日任务状态管理能力。
* 手动触发复盘能力。

#### 4.3 AI 导师 (Chat Tab)

**用户目标**

通过自然语言向 AI 提问，获得学习解释、计划调整建议和复盘追问。

**功能需求**

* 对话视图区：以气泡聊天流展示历史消息。
* 输入交互区：支持多行文本输入与发送。
* 快捷指令区：提供“调整今日剩余计划”、“解释某个知识点”、“针对昨晚复盘提问”等快捷入口。
* 计划变更反馈：当聊天触发计划调整时，客户端展示调整结果并刷新 Today Tab。
* 上下文提示：对话可关联今日计划、近期复盘报告和用户学习目标。

**依赖的后端能力**

* AI 聊天能力。
* 计划变更意图识别能力。
* 今日计划局部更新能力。
* 对话历史查询与持久化能力。

#### 4.4 知识复盘 (Review Tab)

**用户目标**

查看 Git 学习记录同步状态、AI 复盘结论、遗漏知识点和近期学习活跃度。

**功能需求**

* 同步状态卡片：展示最后一次成功同步时间、当前分支、最新 Commit Hash 摘要。
* AI 诊断报告：展示复盘结论、完成度、执行偏差、遗漏知识点和笔记优化建议。
* 学习轨迹热力图：以类似 Git 贡献图的形式展示近 30 天学习活跃度。
* 历史复盘入口：支持查看最近若干天的复盘摘要。
* 只读复盘约束：首版只展示分析结果，不自动写入用户仓库。

**依赖的后端能力**

* Git 仓库同步与 Diff 分析能力。
* 复盘报告生成与查询能力。
* 学习活跃度统计能力。

#### 4.5 系统设置 (Settings Tab)

**用户目标**

管理学习目标、提醒时间、复盘时间、仓库授权与账号安全配置。

**功能需求**

* 核心目标配置：支持修改全局学习目标、目标周期、每日可投入时间。
* 自动化时间轴配置：设置早间计划生成/提醒时间与夜间复盘触发时间。
* 凭证管理中心：查看仓库授权状态，更新 Token 或重新授权。
* 账号安全：展示绑定手机号，支持退出登录。
* 风险提示：对于可能影响 AI 后续计划的配置变更，展示明确确认提示。

**依赖的后端能力**

* 用户配置查询与更新能力。
* 提醒时间与复盘时间配置能力。
* Git 授权更新能力。
* 登录态失效与退出能力。

### 五、后端能力边界

本文档不定义具体 REST URL，仅约定客户端需要后端提供以下能力。

#### 5.1 用户与认证能力

* 发送手机号验证码。
* 校验验证码并签发客户端登录态。
* 查询当前用户基础信息。
* 退出登录或使当前登录态失效。

#### 5.2 学习目标与用户配置能力

* 保存和查询用户学习目标。
* 保存和查询每日可投入时间、计划提醒时间、复盘触发时间。
* 支持目标变更后影响后续计划生成。

#### 5.3 今日计划与任务能力

* 基于学习目标、历史计划、复盘结果生成每日计划。
* 查询今日计划与任务列表。
* 更新任务状态。
* 支持聊天或手动操作触发计划调整。

#### 5.4 AI 聊天能力

* 接收用户消息并返回 AI 回复。
* 将用户目标、今日计划、近期复盘作为上下文。
* 识别计划变更意图，并返回计划是否被更新的结果。
* 持久化对话历史，支持客户端恢复上下文。

#### 5.5 Git 同步与复盘能力

* 校验 Git 仓库授权状态。
* 拉取默认分支 Commit、文件变更和当日 Diff。
* 过滤与学习无关或过大的文件变更。
* 将 Git 变更与当日计划输入 LLM，生成复盘报告。
* 首版只生成只读复盘报告，不自动提交任何仓库变更。

#### 5.6 短信与定时任务能力

* 发送登录验证码。
* 在用户设定时间触发计划生成与提醒。
* 在用户设定时间触发夜间复盘。
* 短信内容需满足服务商模板审核要求，详细计划内容优先在 App 内展示。

### 六、关键业务数据流

| 业务场景 | 触发时机 | 执行逻辑与预期产出 | 客户端职责 |
| --- | --- | --- | --- |
| 手机号登录 | 用户主动触发 | 后端发送并校验验证码，签发登录态 | 展示输入、倒计时、错误提示与登录结果 |
| 初始化配置 | 首次登录后 | 后端保存学习目标与 Git 仓库配置，并校验仓库可访问性 | 引导用户填写目标、仓库地址和授权信息 |
| 早晨计划流 | 定时器触发或用户手动刷新 | 后端调用 LLM 生成当日计划，入库持久化，并按配置发送提醒 | 展示今日计划、任务状态和生成时间 |
| 聊天调度流 | 用户发送消息 | 后端返回 AI 回复；若识别为计划变更，则更新当日计划 | 展示回复，必要时刷新 Today Tab |
| 手动复盘流 | 用户提前结束学习 | 后端拉取当前 Git 变更并生成阶段性复盘 | 展示同步状态与复盘报告 |
| 夜间复盘流 | 定时器触发 | 后端拉取当日 Git Diff，与计划对比后生成复盘报告，并影响次日计划 | 展示最新复盘结果与学习活跃度 |
| 知识库改进建议 | 远期能力 | AI 生成笔记修改建议或 PR，用户确认后生效 | 展示建议、差异和确认入口 |

### 八、阶段规划

#### Phase 1：客户端页面框架与 Mock 数据

* 搭建 Onboarding、Today、Chat、Review、Settings 页面结构。
* 使用 Mock 数据完成核心页面展示。
* 完成底部导航、基础状态交互和页面跳转。
* 不接入真实后端，不处理真实 Git 同步。

#### Phase 2：基础后端接入

* 接入手机号验证码登录。
* 接入学习目标、用户配置、今日计划、任务状态接口。
* 接入 AI 聊天基础能力。
* 支持聊天触发今日计划调整。

#### Phase 3：Git 只读复盘

* 接入 Git 仓库授权校验与同步状态展示。
* 接入只读复盘报告。
* 展示近 30 天学习活跃度。
* 不自动写入用户仓库。

#### Phase 4：自动化提醒与夜间复盘

* 接入早间计划生成与短信提醒。
* 接入夜间自动复盘。
* 客户端展示自动化状态、失败提示和最近执行结果。

#### Phase 5：知识库改进建议与用户确认

* AI 生成笔记补充建议、文件修改建议或 PR。
* 用户在客户端确认后才允许后端执行写入或提交动作。
* 展示修改原因、影响范围和变更差异。

### 九、风险与约束

#### 9.1 Token 与凭证安全

Git Token、LLM Key、短信服务密钥必须由后端安全存储。客户端只展示授权状态，不长期保存敏感凭证明文。

#### 9.2 自动写仓库风险

AI 自动 Commit 并 Push 到用户仓库不作为首版能力。远期如支持写入，必须经过用户确认，优先采用生成建议或 PR 的方式。

#### 9.3 短信模板限制

国内短信服务通常要求模板审核，不适合发送完全自由的 AI 文本。短信首版应只发送固定模板提醒，详细计划和复盘内容在 App 内展示。

#### 9.4 LLM 上下文限制

Git Diff 可能超出模型上下文限制。后端需要过滤无关文件、限制文件大小、分块摘要，并只把与学习目标相关的内容输入模型。

#### 9.5 复盘准确性

AI 复盘结论可能存在误判。复盘报告应以建议和辅助判断形式呈现，不应作为对用户学习成果的绝对评价。

---

### 十、验收标准

* 文档读者能够明确知道当前仓库只负责 HarmonyOS 客户端，后端由独立工程建设。
* 文档支持后续拆出每阶段 PRD Plan，不会将完整愿景误解为一次性开发范围。
* 每个客户端页面都有明确的用户目标、展示内容、交互入口和依赖的后端能力。
* 后端职责以能力边界描述，不陷入具体 REST URL 或字段全集设计。
* 高风险自动化能力有明确约束，不默认执行破坏性写操作。
* 首版可落地闭环清楚：登录 -> 目标配置 -> 今日计划 -> AI 聊天 -> 只读复盘 -> 次日调整。
